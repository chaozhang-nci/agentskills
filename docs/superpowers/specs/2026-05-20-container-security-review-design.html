01

Problem

CVE findings in container images are currently caught server-side — in CI/CD builds or Twistlock/Prisma scans. By that point the feedback loop is slow and expensive to fix.

This skill shifts that check left: run locally before push, triage immediately, confirm before acting.

02

Scope

In Scope

✓ Docker image CVE scanning
✓ Auto-detect installed scanners
✓ Multi-scanner result merging
✓ Severity-based triage
✓ Prioritised action plan
✓ Confirm-before-fix workflow

Out of Scope

✗ Filesystem / IaC scanning
✗ Source code SAST
✗ Automated deployment
✗ CI/CD pipeline integration
✗ Kubernetes manifest scanning

04

Scanner Detection

Probes for installed scanners in order. Runs all found unless one is specified. Click "Simulate Detection" to see how auto-detect works.

Scanner
Detection
Scan Command
Trivy
trivy version
trivy image --format json <image>
Snyk
snyk version
snyk container test <image> --json
Grype
grype version
grype <image> -o json
Docker Scout
docker scout version
docker scout cves <image> --format json
Twistlock / Prisma
twistcli version
twistcli images scan --address $PRISMA_URL <image>
●

Multi-scanner merge: deduplicate findings by CVE ID. If the same CVE appears in multiple scanner outputs, use the highest severity reported and note which scanners flagged it.

07

Fix Behaviour

OS / Base-image fixes

1
Update Dockerfile
Replace FROM line with confirmed candidate base image tag
2
Rebuild Image
docker build -t <image> .
3
Rescan
Run same scanner(s) to confirm the CVE is absent
4
Report Result
Fixed / still present / newly introduced

App dependency fixes

1
Apply Package Upgrade
Run appropriate package manager command for the ecosystem
2
Rebuild if Needed
Rebuild image to pick up updated dependencies
3
Rescan & Report
Confirm CVE is gone; report result

Hard Stops — do not attempt fix if:

  • No Dockerfile is accessible for OS fixes
  • No fixed version exists for a CVE
  • Scanner rescan cannot be run after the fix
